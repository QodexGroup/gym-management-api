# Membership Plans & Billing — Industry Benchmark and Recommended Policies

*Purpose: benchmark how established gym / fitness management platforms model client membership plans and recurring billing, and recommend proper default policies for GymHubPH (a cash‑heavy Philippine market, not card‑on‑file). This is a recommendation document — no code changes implied.*

---

## 1. Executive summary

The single biggest finding: **every serious platform (and Stripe, which most of them build on) separates the recurring *agreement* from the *bills* it produces.** They never infer the billing cycle from a date field the way our current code does — the cycle is a real record.

The canonical four‑layer model is:

1. **Plan / Price** — the catalog template (amount, interval, entitlements).
2. **Subscription / Membership / Contract** — the recurring agreement instance (customer + plan + status + cycle anchor + current period). Holds **no money**.
3. **Invoice / Bill** — the per‑period bill generated by the agreement (line items, amount due/paid, period start/end, its own status).
4. **Payment / Charge** — the money movement that settles an invoice.

This is exactly the "period/invoice ledger" direction proposed earlier — the benchmark **confirms** it is the industry norm, not an over‑engineered idea. Stripe's own objects are `Subscription → Invoice → Invoice line item → PaymentIntent`, and PushPress/Glofox/Mindbody/Zen Planner all mirror the split ("membership/contract" vs "autopay/invoice/payment").

Our current implementation collapses layers 2–3 (a single mutable `CustomerMembership` + `CustomerBill` inferred from `bill_date`), which is the root of the edge cases we've hit.

## 2. Benchmark comparison (by dimension)

| Dimension | Industry norm (Stripe + platforms) | GymHubPH today | Gap? |
|---|---|---|---|
| **Agreement vs bill** | Distinct objects; bill linked to the period it covers | One membership row + bills inferred by date | **Yes** |
| **Statuses** | pending, active, past_due/overdue, frozen/paused, cancelled, expired | active, expiring, expired, deactivated | Missing **past_due**, **frozen** |
| **Billing anchor** | Anniversary (join date) *or* fixed calendar day; configurable | Anniversary (bill_date + plan period) | OK (anniversary is fine) |
| **Failed payment** | Auto card retries + dunning (Stripe 8×/2wk; Glofox 4× at +0/+1/+4/+11 days); "past due"; access cut after ~7 days | N/A (cash) — expiry check only | Partial (see §4) |
| **Partial payment** | Autopay is **all‑or‑nothing**; partials only on manual/POS invoices with a tracked remaining balance | **Any** payment (even partial) extends a **full** period | **Yes — too loose** |
| **Freeze / hold / pause** | Universal, first‑class, date‑ranged, auto‑resume; billing deferred (push expiry) or reduced‑price | **Does not exist** | **Yes — missing feature** |
| **Reactivation** | Soft‑lapse = *resume* in place; full lapse = *restart* new sign‑up; join fee usually **waived**, rarely mandatory | Mandatory reactivation fee + hardcoded 1 free month | **Yes — idiosyncratic** |
| **Plan change (proration)** | Stripe prorates by default; PushPress offers "Now (prorated)" vs "Next bill date"; Glofox/Zen ≈ cancel+re‑add | Change/re‑bill, no proration model | Partial |

## 3. The data model, as the industry does it

**Statuses to adopt (superset seen across PushPress "Paused/Alert/Pending/Cancelled/Completed", Glofox "Active/Paused/Overdue", Mindbody "Suspend/Expired", Stripe "trialing/active/past_due/canceled/unpaid/paused"):**

- `pending` / `upcoming` — starts on a future date, not yet billing.
- `active` — current period paid (or within grace).
- `past_due` — a period bill is unpaid past its due date but still within grace (member may keep limited access per policy).
- `frozen` / `on_hold` — intentionally paused for a date range (see §5.3).
- `expired` — lapsed, no active period.
- `cancelled` — deliberately terminated.

**Invoice statuses** (Stripe: `draft/open/paid/uncollectible/void`) map cleanly to our current `active/partial/paid/voided` — keep those, they're fine.

**Key move:** a membership bill should carry an explicit `period_start`, `period_end`, `cycle_sequence`, and `subscription_id`, with a **unique constraint** so the "same period" can't be double‑billed (the doc already warns manual+automated bills can duplicate). Then "previous cycle" is `sequence < current`, not a fragile date comparison.

## 4. The four policy questions — recommended defaults (benchmark‑backed)

### Q1. Does a partial payment activate the full period?
**Industry:** No. Automatic membership charges are **all‑or‑nothing** everywhere (Stripe explicitly does not support partial payment on auto‑charged subscription invoices). Partial payments exist only on **manual/POS invoices**, where the invoice shows `partially paid` with a tracked `amount_remaining` (PushPress: "collect a partial payment and leave the rest open"; Gymdesk: multiple partial payments per charge).

**Recommended default:** A membership **period is "paid" (coverage granted) only when its bill is fully paid.** A partial payment reduces the balance and is tracked on AR, but does **not** by itself extend membership. Offer an optional per‑gym toggle "activate on deposit" for gyms that want to grant access on a down‑payment — but default to full‑payment‑activates. This directly fixes today's "pay ₱1 → full month" looseness.

### Q2. Grace period / dunning?
**Industry:** Card platforms retry (Stripe default **8 retries over 2 weeks**; Glofox **4 attempts at +0/+1/+4/+11 days**), mark `past_due`, and **cut access after ~7 days** overdue. In a cash market there are no card retries, so "dunning" becomes **reminders**, not retries.

**Recommended default:** Add a `past_due` state with a **configurable grace period (default 7 days)** after a period's due date. During grace: membership stays usable (or limited) and the member gets reminder notifications (reuse the existing expiring‑membership notifications). After grace with no payment → `expired` and access blocked. Access‑cutoff day should be a setting.

### Q3. Reactivation / win‑back?
**Industry:** A member who lapsed for **non‑payment** is **resumed in place** (PushPress "Activate plan," Mindbody "Activate"). A member who **fully left** is **restarted as a fresh sign‑up**. **No platform mandates a reactivation fee** — the join/setup fee is a configurable one‑time charge that operators commonly **waive** as a win‑back incentive (reactivating a former member is ~5–7× cheaper than new acquisition; waive within ~90–180 days).

**Recommended default:** Make the reactivation fee **optional/configurable**, not mandatory, and make the "free month" a **promo‑period flag** rather than hardcoded 1 month. Distinguish **resume** (recent lapse, same plan, continue the agreement) from **restart** (new agreement, possibly new plan). Our current forced "reactivation fee + 1 free month regardless of plan" is out of step with the benchmark.

### Q4. Proration on mid‑cycle plan change (upgrade/downgrade)?
**Industry:** Stripe prorates by default (credit unused old + charge remaining new). **PushPress** exposes the cleanest operator choice: **"Now"** (immediate, prorated) vs **"Next bill date"** (change takes effect at renewal, no proration). Glofox and Zen Planner largely handle it as cancel‑and‑re‑add with limited/manual proration (Zen forces prorate‑to‑the‑1st).

**Recommended default:** Offer the PushPress‑style toggle, and **default to "change at next renewal" (no proration)** — simplest and least error‑prone for a cash gym. Offer "change now (prorated)" as an option for gyms that want it. Snapshot the plan price on each period so historical bills keep their original amount when plans/prices change.

## 5. Benchmark‑driven additions we're currently missing

### 5.1 Explicit billing period / cycle record
Confirmed as the industry standard (Stripe `current_period_start/end`, `billing_cycle_anchor`; every platform bills against a defined period). Adopt `period_start/end/sequence/subscription_id` on bills → kills duplicate bills, makes "previous cycle" trivial and reliable, and enables clean history and aging.

### 5.2 Billing anchor
Both anniversary (join date) and fixed calendar day (e.g., the 1st) are legitimate and configurable across the industry. **Anniversary is simpler for a cash gym** because it avoids first‑period proration entirely — keep our current anniversary approach; just make it explicit on the subscription (`billing_anchor_day`).

### 5.3 Freeze / Hold / Pause — **the biggest missing feature**
Universal and first‑class in **every** platform we checked (PushPress *Paused*, Gymdesk *Frozen*, Glofox *Paused* — "sometimes called hold/freeze/suspend", Mindbody *Suspend*, TeamUp *Hold*, Zen Planner *Hold*). Common design:
- **Date‑ranged** (freeze from → to), schedulable in advance, **auto‑resumes** on the end date.
- Two billing models: **(a) defer/extend** — no charge during the freeze and the **expiry/next‑bill date is pushed out by the freeze length** so paid time isn't lost (PushPress, Glofox, Gymdesk "extend by freeze duration", Mindbody suspend); **(b) reduced‑price freeze** — a lower charge continues (Mindbody Booker freeze; PushPress optional Hold Fee).
- Min/max durations and per‑year limits are **gym‑configurable policy**, not platform‑enforced.

**Recommendation:** add a `frozen` status with a start/end date and, as the default, the **defer/extend model** (push `membership_end_date` out by the frozen days). Optionally support a freeze fee. This is a genuinely new capability members will expect.

### 5.4 Cash / manual payments, AR aging, and account credit
Our invoice‑first model is already right (matches PushPress). Add:
- A first‑class **manual/offline payment type** (cash, GCash, Maya, bank transfer) that staff mark as paid — never gate features behind a card processor (Gymdesk "Manual Payment"; PushPress shows only Cash/Check/Comp when no card on file).
- **Partial settlement with a tracked remaining balance** on the invoice.
- **AR aging** buckets (current / 30 / 60 / 90 days) and a per‑member + gym‑wide "money owed" report (Mindbody Account Balances/Unpaid Visits; Glofox "Money Owed").
- An optional **account credit / prepaid wallet** that may go negative ("pay next visit") gated by staff permission (Glofox Account Balance is the reference model), usable to settle future invoices.
- Keep a strong **audit trail** on manual payments (who marked paid, when, reference/OR number) — the weak point of cash operations.

### 5.5 Philippines‑specific must‑haves
Independently surfaced in the research and worth designing in from the start: **BIR Official Receipt (OR) numbers**, **12% VAT** handling, mandatory **20% senior‑citizen (RA 9994)** and **20% PWD (RA 10754)** discounts, e‑wallet/bank rails (**GCash, Maya, InstaPay/PESONet, OTC**), and **installment** options for annual memberships.

## 6. How this maps to the earlier proposal

The benchmark validates the earlier **Subscription + MembershipPeriod + Invoice** ledger recommendation and its phased rollout:

- **Phase 0** (add `period_start/end/sequence/subscription_id` + unique index to bills) is low‑risk and already kills duplicate bills and the fragile "previous cycle" date logic.
- **Phase 1–2** introduce the subscription/period tables and move renewal, freeze, reactivation, and expiration onto periods.
- New, benchmark‑driven scope to fold in: **`past_due` + grace**, **`frozen`/freeze with expiry‑extension**, **full‑payment‑activates policy**, **optional/configurable reactivation fee + promo period**, **AR aging + credit wallet**, and **PH compliance (OR/VAT/senior‑PWD)**.

---

## 7. Suggested defaults, at a glance

| Decision | Recommended default | Configurable? |
|---|---|---|
| Partial payment | Coverage only on **full** payment; balance tracked on AR | Yes ("activate on deposit" toggle) |
| Grace period | **7 days** `past_due` before expiry/access cutoff | Yes (days) |
| Reactivation fee | **Off** by default; free/promo period is a flag | Yes |
| Plan change | **At next renewal** (no proration) | Yes ("change now, prorated") |
| Billing anchor | **Anniversary** (join date) | Yes (fixed day) |
| Freeze | Enabled; **defer/extend** expiry by frozen days | Yes (freeze fee, max length) |
